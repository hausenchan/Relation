## 一、业务模式与定位

### 1.1 业务模式

你们是**广告联盟/网盟业务**：

```
上游预算方（优量汇/穿山甲/广点通）
         ↓ 商务BD接入
    【预算池】
         ↓ 加工
【产品/任务广告】（小程序/APP）
         ↓ 分发
    ┌─────────┴─────────┐
    ↓                   ↓
【SDK对接媒体】      【买量投放】
爱奇艺极速版等        巨量引擎等
    ↓                   ↓
  自然流量            付费流量
    └─────────┬─────────┘
              ↓
         【收益结算】
```

### 1.2 系统定位

**核心定位**：团队协作效率提升系统

**不做**：
- ❌ 数据分析、实时监控（已有数据后台）
- ❌ 财务结算、账期管理
- ❌ 复杂的BI报表

**要做**：
- ✅ 流程协作、任务流转
- ✅ 进度跟踪、目标管理
- ✅ 周报管理、工作沉淀
- ✅ 部门协作接口定义

## 二、核心业务流程

### 2.1 业务协作流（部门协作）

```
商务部 → 产运部 → 研发部
  ↓        ↓        ↓
线索    策略需求   开发任务
```

**详细流程**：

```
线索来源（多源头）
  ├─ 商务发现 → 预算方/媒体线索
  ├─ 产运发现 → 公司研究/数据分析
  └─ 用户反馈 → 产品优化线索
         ↓
    【线索池】（统一管理）
         ↓
    产运评估 → 制定策略
         ↓
    策略执行（两条路径）
    ┌─────────┴─────────┐
    ↓                   ↓
【运营执行】        【研发需求】
AB测试/配置调整    需要开发的功能
    ↓                   ↓
  效果追踪          开发上线
    └─────────┬─────────┘
              ↓
         反馈闭环
```

### 2.2 目标管理流（管理视角）

```
老板制定公司目标
    ↓
部门组长制定部门目标
    ↓
个人执行（业务流转）
    ↓
周报更新目标进度
    ↓
目标达成度追踪
```

## 三、完整菜单架构

```
🏠 工作台
   ├── 我的任务（待办 + 已指派）
   ├── 我的目标（快捷查看）
   ├── 我的周报（组长 + 被指定成员可见）
   ├── 待审批（送礼/出差）
   └── 提醒事项

🎯 目标与计划
   │
   ├── 目标管理
   │    ├── 目标列表（年度/季度/月度/周）
   │    └── 目标看板（按部门分组展示进度）
   │
   └── 周报管理
        ├── 我的周报（组长 + 被指定成员）
        ├── 周报汇总（老板视角）
        └── 周报人员管理（老板指定谁写周报）

📋 业务流转
   │
   ├── 线索池
   │    ├── 我提交的线索
   │    ├── 待评估线索（产运视角）
   │    └── 全部线索（Leader视角）
   │
   ├── 策略管理（产运核心）
   │    ├── 策略列表（全部策略）
   │    ├── 预算方策略（预算交付岗位视角）
   │    │    ├── 预算方列表
   │    │    └── 策略时间线
   │    ├── 媒体策略（流量运营岗位视角）
   │    │    ├── 媒体列表
   │    │    ├── 月度规划
   │    │    └── 全局规划
   │    ├── 运营执行中（AB测试/配置调整）
   │    ├── 研发需求中（已提交研发的）
   │    └── 策略效果追踪
   │
   └── 研发任务（研发部）
        ├── 需求列表
        ├── 开发进度
        └── 上线记录

🤝 商务协作
   ├── 人脉管理（上游预算方/下游媒体联系人）
   ├── 互动记录（拜访/会议记录）
   ├── 公司研究（潜在合作方调研）
   ├── 人才库（Talents）
   └── 客户管理（Clients）

👥 团队管理
   ├── 任务看板（Leader视角，看全组任务）
   ├── 送礼管理（申请/审批/记录）
   └── 出差管理（申请/审批/费用）

⚙️ 系统管理
   ├── 用户管理
   ├── 小组管理
   └── 菜单权限
```

## 四、核心模块详细设计

### 4.1 线索池模块

**功能定位**：统一管理所有部门提交的业务线索

**线索表结构**：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| 线索标题 | TEXT | "字节穿山甲游戏预算" |
| 线索类型 | TEXT | 预算方/媒体/产品优化/其他 |
| 来源部门 | TEXT | 商务部/产运部/研发部 |
| 来源方式 | TEXT | 商务BD/公司研究/数据分析/用户反馈 |
| 提交人 | INTEGER | 用户ID |
| 提交时间 | DATETIME | 2026-04-10 |
| 线索描述 | TEXT | 详细说明 |
| 关联公司 | INTEGER | 可关联公司研究模块 |
| 关联人脉 | INTEGER | 可关联人脉管理 |
| 优先级 | TEXT | P0/P1/P2 |
| 评估状态 | TEXT | 待评估/评估中/已采纳/已拒绝 |
| 评估人 | INTEGER | 产运Leader |
| 评估结论 | TEXT | 文本说明 |
| 关联策略 | INTEGER | 如果采纳，关联到哪个策略 |

**权限设计**：
- 商务/产运/研发都能提交线索
- 只有产运能评估线索
- Leader能看到所有线索

**关键操作流程**：
```
1. 任何部门提交线索
2. 产运Leader收到通知
3. 产运评估线索（采纳/拒绝）
4. 如果采纳 → 创建策略
5. 如果拒绝 → 记录原因，通知提交人
```

### 4.2 策略管理模块

**功能定位**：产运部核心工作台，管理所有商业化策略

#### 4.2.0 产运部门岗位说明

产运部门有两个岗位，分别负责不同的策略管理：

**1. 预算交付岗位**
- 职责：对单一联盟预算做商业化优化
- 对象：单个预算方（如优量汇、穿山甲、广点通）
- 视角：针对某个预算方的深度优化
- 策略类型：预算方策略

**2. 流量运营岗位**
- 职责：对N多个媒体APP做全局和月度策略规划
- 对象：多个媒体APP（如爱奇艺极速版、快手极速版等）
- 视角：全局规划 + 月度规划
- 策略类型：媒体策略

**策略三个维度（统一链路）**：
1. **变现维度**：如何提升变现效率（eCPM、ARPU）
2. **流量维度**：如何获取和优化流量（DAU、留存）
3. **链路维度**：如何优化整个链路（填充率、转化率）

**策略表结构**：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| 策略标题 | TEXT | "优量汇eCPM提升策略" |
| **策略对象类型** | TEXT | **预算方策略/媒体策略** |
| **关联预算方** | INTEGER | 如果是预算方策略，关联预算方ID |
| **关联媒体** | INTEGER | 如果是媒体策略，关联媒体ID（NULL表示全部媒体） |
| **策略维度** | TEXT | **变现/流量/链路** |
| **策略规划类型** | TEXT | **单次策略/月度规划/全局规划** |
| **规划月份** | TEXT | 如果是月度规划，记录月份（2026-05） |
| 关联线索 | INTEGER | 来自哪个线索 |
| 策略描述 | TEXT | 详细方案 |
| 策略目标 | TEXT | 提升eCPM 20% |
| 执行方式 | TEXT | **运营执行/研发需求** |
| 策略状态 | TEXT | 待执行/执行中/已完成/已暂停 |
| **负责岗位** | TEXT | **预算交付/流量运营** |
| 负责人 | INTEGER | 产运人员ID |
| 创建时间 | DATETIME | 2026-04-12 |
| 关联目标 | INTEGER | 关联部门目标 |

#### 4.2.1 运营执行（不需要研发）

**运营执行记录表**：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| 关联策略 | INTEGER | 属于哪个策略 |
| 执行类型 | TEXT | AB测试/配置调整/内容优化 |
| 执行内容 | TEXT | "调整广告位从3个增加到5个" |
| 测试版本 | TEXT | A版本/B版本 |

| 是否采用 | INTEGER | 1=采用，0=回滚 |
| 执行人 | INTEGER | 产运人员ID |

**典型场景**：
- AB测试：两个版本对比效果
- 配置调整：修改广告位数量、刷新频率
- 内容优化：调整任务奖励、UI文案

**需求提交表**：

| 字段 | 类型 | 说明 |
|------|------|------|


### 4.5 周报管理模块

#### 4.5.1 周报权限规则

|------|----------|

| 普通成员/Member | **默认不需要**填写周报，但老板可以指定某些人写（个人周报） |

#### 4.5.2 周报表结构

| boss_comment_time | DATETIME | 评论时间 |
| created_at | DATETIME | 创建时间 |

#### 4.5.3 周报目标关联表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| report_id | INTEGER | 周报ID |
| goal_id | INTEGER | 目标ID |
| last_week_progress | TEXT | 上周进度 |
| this_week_progress | TEXT | 本周进度 |
| progress_percent | INTEGER | 进度百分比 |
| progress_note | TEXT | 进度说明 |

#### 4.5.4 用户表增加字段

| 字段 | 类型 | 说明 |
|------|------|------|
| need_weekly_report | INTEGER | 是否需要写周报（0=不需要，1=需要） |

**默认值**：
- 组长（role='leader'）：need_weekly_report = 1
- 普通成员（role='member'）：need_weekly_report = 0
- 老板可以手动修改任何人的这个字段

#### 4.5.5 周报内容差异

**组长周报**：
- 团队工作总结
- 团队成果（可关联线索/策略/需求）
- 团队目标进度更新
- 下周团队计划
- 团队遇到的问题
- 需要的支持

**成员周报**（被指定的普通成员）：
- 个人工作总结
- 个人完成事项（可关联线索/策略/需求）
- 个人目标进度更新
- 下周个人计划
- 遇到的问题
- 需要的支持

**老板功能**：
- 查看所有周报（组长 + 被指定成员）
- 评论周报

### 13.2 角色设计（三层）

| 角色代码 | 中文名 | 谁使用 | 核心权限 |
|----------|--------|--------|----------|
| `admin` | 管理员 | CEO / COO / CTO / CMO | 看全部数据、给任何人派活、管系统配置 |
| `leader` | 负责人 | 各小组独立leader | 看本组数据、给本组成员派活、写周报 |
| `member` | 普通成员 | 广告运营 / 开发 / 测试 | 看自己任务、完成任务 |

> **注**：CMO 和 CTO 是 `admin` 角色，天然拥有给全员派活的权限，无需单独设为小组 leader。

---

### 13.3 账号设计

#### 四个老板账号

| 账号 | 角色 | 说明 |
|------|------|------|
| CEO | `admin` | 看全部数据，无需管系统配置 |
| COO（你） | `admin` | 超管，兼管系统配置（创建用户/分配角色/配置权限） |
| CTO | `admin` | 看全部数据，直接管研发所有成员 |
| CMO | `admin` | 看全部数据，直接管整个商务部 |

#### 超管账号 vs 老板账号的区别

| 维度 | 超级管理员（COO兼任） | 其他老板账号（admin） |
|------|----------------------|-----------------------|
| 定位 | 系统维护 + 业务管理 | 纯业务管理 |
| 创建用户 | ✅ 可以 | ✅ 可以（admin均可） |
| 配置菜单权限 | ✅ 可以 | ✅ 可以（admin均可） |
| 查看全部数据 | ✅ | ✅ |
| 给任何人派活 | ✅ | ✅ |
| 实际区别 | 建议由你（COO）负责系统维护，其他人不用管 | 专注业务使用即可 |

> **结论**：四个老板都是 `admin` 角色，权限完全相同。COO（你）负责实际的系统维护工作（创建账号、配置权限等），其他三位老板不用关心系统配置，专注业务即可。

### 13.8 特殊情况处理

#### CMO兼任预算商务小组leader
- CMO 是 `admin` 角色，天然可以给预算商务小组所有成员派活
- 预算商务小组**不需要设独立 leader**
- 小组 leader 字段可留空，或直接指向 CMO 账号
- 系统里 CMO 能看到该小组所有成员的任务和数据

#### CTO管研发（扁平化）
- CTO 是 `admin` 角色，天然可以给研发所有成员派活
- 研发组**不需要设独立 leader**
- 所有开发/测试都是 `member` 角色，直接由 CTO 管理
- 后续如果研发团队扩大，需要设 leader 时，直接在系统里调整即可

#### COO管产运
- COO（你）是 `admin` + 超管
- 产运各小组有独立 leader（`leader` 角色），向你汇报
- 你可以看到产运所有数据，也可以直接给任何产运成员派活

#### 4.2.7 业务流程调整

**预算交付岗位工作流**：

```
1. 接收线索（预算方线索）
2. 评估线索 → 采纳
3. 创建策略
   - 策略对象类型：预算方策略
   - 关联预算方：优量汇
   - 策略维度：变现/流量/链路（选一个）
   - 规划类型：单次策略
   - 负责岗位：预算交付
4. 执行策略（运营执行 OR 研发需求）
5. 追踪效果
6. 周报更新进度
```

**流量运营岗位工作流（月度规划）**：

```
1. 每月20日左右，创建下月规划
   - 策略对象类型：媒体策略
   - 关联媒体：全部媒体 OR 具体媒体
   - 策略维度：流量（通常）
   - 规划类型：月度规划
   - 规划月份：2026-05
   - 负责岗位：流量运营

2. 在月度规划下，创建具体策略
   - 每个媒体的具体优化策略
   - 新媒体接入计划

3. 按周执行策略

4. 月末总结效果

5. 周报更新进度
```

**流量运营岗位工作流（单次策略）**：

```
1. 发现某个媒体的具体问题
2. 创建单次策略
   - 策略对象类型：媒体策略
   - 关联媒体：爱奇艺极速版
   - 策略维度：链路（如填充率问题）
   - 规划类型：单次策略
   - 负责岗位：流量运营

3. 执行策略

4. 追踪效果
```
